# carina#3400: WIT async export design

Status: design only (2026-06-06)
Parent: [carina#3400 root-cause investigation](./2026-06-06-3400-plan-hang-multiprovider-assume-role-design.md)

## Goal
<!-- derived-from ./2026-06-06-3400-plan-hang-multiprovider-assume-role-design.md -->

Make the carina WASM provider boundary unable to run `wasi:http/outgoing-handler` from a synchronous component export that waits with `subscribe().block()` before returning. The implementation should move every provider operation that can perform external I/O to async-lifted WIT exports, update the guest SDK and host call sites to the generated async shape, and leave provider authors with one clear async trait surface that maps to the existing core provider futures without an embedded guest runtime.

## Non-goal

Do not redesign the provider protocol, state model, AWS assume-role configuration syntax, or provider registry resolution. This change is not a wasmtime bump-only fix, not a retry/timeout workaround, and not a special case for the five-provider registry-dev stack; it is a boundary-shape change that removes the invalid sync-export-calls-async-import pattern.

## Affected components inventory

`carina-plugin-wit/wit/provider.wit` is the root schema change. `initialize`, `read`, `read-data-source`, `create`, `update`, and `delete` change from plain `func` to `async func`. `info`, `schemas`, `provider-config-attribute-types`, `validate-config`, `provider-config-completions`, `identity-attributes`, `validate-custom-type`, `get-enum-aliases`, `normalize-desired`, `normalize-state`, `hydrate-read-state`, and `merge-default-tags` stay sync unless the implementation experiment shows the generated binding shape forces whole-interface async. The long-running set is the set that can use external provider APIs or waiters.

`carina-plugin-wit/wit/world.wit` keeps both worlds, `carina-provider` and `carina-provider-with-http`, exporting the same `provider` interface. No new world is required for the recommended path. If the compatibility fallback is chosen after experiments, this file would instead export both `provider` and `provider-async`; that is not the recommended design.

`carina-plugin-sdk/src/lib.rs` changes the `CarinaProvider` trait shape. `initialize`, `read`, `read_data_source`, `create`, `update`, and `delete` become async-returning methods. The sync metadata and validation methods stay sync. The module should expose a SDK-local boxed future alias for these methods instead of requiring every provider crate to depend on `async-trait`.

`carina-plugin-sdk/src/wasm_guest.rs` changes the `export_provider!` macro. The generated `impl exports::carina::provider::provider::Guest for WasmGuest` must implement async methods for the async WIT exports, convert WIT arguments before awaiting, call the new async trait methods, and convert results after await. The current `get_provider() -> &'static std::sync::Mutex<T>` is no longer acceptable for awaited methods because it would hold a blocking mutex guard across await. Use a single-thread-friendly async-aware state wrapper for operation serialization, or restructure the generated adapter so provider state is temporarily moved out, awaited, and moved back without a blocking guard.

`carina-plugin-sdk/src/wasi_http.rs` is not the main API surface, but it is the guest import path that currently makes the invalid pattern observable. `execute`, `send_request`, `WasiHttpConnector::call`, and `WasiHttpClient` should be rechecked after async exports land. The first implementation can keep the existing synchronous `send_request` adapter if it is called only from an async-lifted export, but the preferred follow-up is to add an async `execute_async` path that awaits the `FutureIncomingResponse` via wit-bindgen async support rather than calling `pollable.block()`.

`carina-plugin-sdk/src/wasi_http_body.rs` should remain behaviorally unchanged. Its body classification, content-length injection, chunk sizing, and stream read/write helpers are transport correctness fixes independent of export syncness. Only function visibility or return plumbing should change if `wasi_http.rs` splits sync and async cores.

`carina-plugin-host/src/wasm_factory.rs` changes all generated call wrappers for async-lifted exports: `WasmBindings::call_initialize`, `call_read`, `call_read_data_source`, `call_create`, `call_update`, and `call_delete`. The implementation must use the wasmtime component async call sequence generated by bindgen, or drop to `Func::call_concurrent` inside `Store::run_concurrent` if the generated helper exposes only the lower-level shape. `create_initialized_instance`, `LockedStore`, `with_operation_timeout`, and the provider method implementations are the major host call sites.

`carina-provider-mock/src/main.rs` is the reference guest implementation shipped with carina. Its `MockProcessProvider::initialize`, `read`, `read_data_source`, `create`, `update`, and `delete` implementations change to the new async-returning trait methods. Existing pure map operations can return ready futures. Any mock HTTP initialization helper used for regression tests should become async-shaped and must not use an embedded runtime.

`carina-provider-aws/carina-provider-aws/src/main.rs` changes `AwsProcessProvider`. Remove the `runtime: tokio::runtime::Runtime` field, remove `AwsProcessProvider::new()`'s runtime construction, and convert `initialize`, `read`, `read_data_source`, `create`, `update`, and `delete` to await the existing async `AwsProvider` and core `Provider` futures directly. The account-guard path remains in `initialize`, but `AwsProvider::new_with_account_guard(...).await` and `provider.verify_account_id().await` run in the exported async task rather than inside `block_on`.

`carina-provider-aws/carina-provider-aws/src/lib.rs` is mostly already async on the core provider side. `AwsProvider::new_with_account_guard`, `build_config`, `wrap_with_assume_role`, and `verify_account_id` keep their async shape. The in-process `AwsProviderFactory::create_provider` already returns `BoxFuture`; it should not need semantic changes, but tests may need updates because the SDK trait signature changes.

`carina-provider-awscc/carina-provider-awscc/src/main.rs` changes the same way as the AWS process provider. Remove `runtime`, convert the WIT-facing `initialize`, `read`, `read_data_source`, `create`, `update`, and `delete` implementations to async-returning methods, and await `AwsccProvider::new_with_config`, core provider operations, and normalizer futures directly where those methods remain part of the async export set.

`carina-provider-awscc/carina-provider-awscc/src/lib.rs` is also mostly ready on the in-process side. `AwsccProviderFactory::create_provider` already returns `BoxFuture`, and `AwsccProvider::new_with_config` already awaits `build_config` and assume-role wrapping. The implementation should only change if SDK trait imports or helper types move.

## WIT-level design choices

Option A marks every long-running export as `async func`: `initialize`, `read`, `read-data-source`, `create`, `update`, and `delete`. This directly matches the failure mode, keeps the WIT interface name stable, and forces every I/O-capable provider operation through the async component-model path. The cost is a breaking WIT and SDK change for all downstream WASM providers. The churn is useful because a provider that still uses the old sync trait will fail at compile time rather than reintroducing the invalid pattern.

Option B adds a parallel `provider-async` interface and keeps `provider` sync as a legacy fallback. The upside is staged migration for old provider components. The downside is two protocol surfaces, two host dispatch paths, more tests, and a standing risk that carina accidentally loads the legacy sync path for an HTTP-capable provider. That leaves the broken state representable. It also complicates provider registry behavior because the host must classify a component's capability at instantiation time and report useful errors when only the old interface is present.

Option C splits exports by I/O cost: long-running operations become async, metadata, schema, validation, completion, identity, enum alias, and pure normalization functions stay sync. This is the smallest WIT change that removes the root cause. It limits host and guest async plumbing to operations that can call provider APIs while preserving cheap sync calls for startup metadata and config validation. The risk is misclassification: a future provider could put network I/O in `validate-config` or a normalizer and recreate the same pattern. That risk should be handled by SDK contract and tests: sync trait methods must be documented and tested as no-I/O methods.

Recommendation: choose Option C now, implemented as an in-place breaking change to the existing `provider` interface. It removes the observed invalid pattern without forcing pure metadata calls through component async, and it avoids a compatibility branch that keeps the bug alive. If the first code experiment shows that wit-bindgen generates a simpler and safer surface when the whole interface is async, switch to Option A before writing provider migrations; do not add Option B unless wasmtime or wit-bindgen cannot support mixed sync/async exports.

Compatibility evidence to verify before coding: carina currently has wasmtime 45 sources available locally and `wasmtime::component::Func::call_concurrent` documents the `Store::run_concurrent` + `Accessor` call shape for async component tasks. The local 45.0.1 source says a task created by `call_concurrent` must be awaited inside `run_concurrent`, and dropping the returned future does not cancel the in-progress wasm task; hard cancellation requires dropping the store. wasmtime 43.0.2 also contains component concurrency APIs, but the parent investigation established that the current sync-export pattern livelocks there, so 43 should not be treated as proof that the async-lifted generated path is sufficient. wasmtime 44 sources are not present in the checked local registry, so 44 support must be verified by experiment, not assumption. wit-bindgen 0.54 is in the carina SDK and local registry sources include async support. The implementation should target the version actually pinned in carina at the time of the PR, currently 45 on this branch, and only claim 44 support after compiling the generated bindings against 44. If 45 supports async lifts but still hangs after the guest no longer blocks, the C-stage verdict changes from "sync export root cause" to "blocking guest wasi-http adapter still present"; then `wasi_http.rs` must move to an awaitable response path in the same fix.

## Guest SDK shape change

Use boxed futures in the public `CarinaProvider` trait, not `async fn` in the trait. The existing carina core provider traits already use `BoxFuture`, provider implementations frequently delegate to those futures, and avoiding `async-trait` keeps the SDK dependency surface smaller for WASM guests. The shape should be:

```rust
pub type BoxFuture<'a, T> = Pin<Box<dyn Future<Output = T> + 'a>>;

fn initialize<'a>(&'a mut self, attrs: &'a HashMap<String, Value>) -> BoxFuture<'a, Result<(), String>>;
fn read<'a>(&'a self, id: &'a ResourceId, identifier: Option<&'a str>, request: ReadRequest) -> BoxFuture<'a, Result<State, ProviderError>>;
fn read_data_source<'a>(&'a self, resource: &'a Resource) -> BoxFuture<'a, Result<State, ProviderError>>;
fn create<'a>(&'a self, id: &'a ResourceId, request: CreateRequest) -> BoxFuture<'a, Result<State, ProviderError>>;
fn update<'a>(&'a self, id: &'a ResourceId, identifier: &'a str, request: UpdateRequest) -> BoxFuture<'a, Result<State, ProviderError>>;
fn delete<'a>(&'a self, id: &'a ResourceId, identifier: &'a str, request: DeleteRequest) -> BoxFuture<'a, Result<(), ProviderError>>;
```

Default implementations for `initialize` can return `Box::pin(async { Ok(()) })`. There should be no default CRUD implementation because each provider must still choose explicit behavior.

The `export_provider!` macro must bridge boxed-future trait methods to async WIT exports. For each async export, convert owned WIT input into protocol-owned values before calling the provider method, call the trait method, await it, then convert the result. For `initialize`, the provider state is mutated; for CRUD it is read-only. Do not hold `std::sync::MutexGuard` across `.await`. A simple safe shape is to keep provider state in a `OnceLock` containing an async mutex that is valid on wasm32, then await the mutex and the provider method inside the async export. That still serializes all operations on one guest instance, matching the host's current one-store-at-a-time behavior, but it does not block the component task with an OS-style mutex. If an async-aware mutex is unavailable for WASM without pulling a runtime, use `RefCell<T>` only after confirming wasmtime never polls two exports from the same instance concurrently; otherwise `RefCell` can panic on reentrant polls.

The macro should keep sync methods sync. `info`, `schemas`, `provider_config_attribute_types`, `validate_config`, `provider_config_completions`, `identity_attributes`, `validate_custom_type`, `get_enum_aliases`, `normalize_desired`, `normalize_state`, `hydrate_read_state`, and `merge_default_tags` may continue to lock with `std::sync::Mutex` only if the async methods use a compatible shared state abstraction. Prefer one state wrapper for all methods so the provider cannot be borrowed by a sync method while an async operation is live.

## Guest provider impl change

AWS `main.rs` should stop acting as a second executor. The process provider becomes a thin protocol adapter:

```rust
struct AwsProcessProvider {
    provider: Option<AwsProvider>,
    normalizer: AwsNormalizer,
}

impl CarinaProvider for AwsProcessProvider {
    fn initialize<'a>(
        &'a mut self,
        attrs: &'a HashMap<String, proto::Value>,
    ) -> BoxFuture<'a, Result<(), String>> {
        Box::pin(async move {
            let core_attrs = convert::proto_to_core_value_map(attrs);
            let region = extract_region(&core_attrs);
            let allowed = extract_string_list(core_attrs.get("allowed_account_ids"));
            let forbidden = extract_string_list(core_attrs.get("forbidden_account_ids"));
            let assume_role = extract_assume_role(core_attrs.get("assume_role"))?;

            let provider = AwsProvider::new_with_account_guard(
                &region,
                allowed,
                forbidden,
                assume_role,
            ).await;
            provider.verify_account_id().await?;
            self.provider = Some(provider);
            Ok(())
        })
    }

    fn read<'a>(&'a self, id: &'a proto::ResourceId, identifier: Option<&'a str>, request: proto::ReadRequest)
        -> BoxFuture<'a, Result<proto::State, proto::ProviderError>>
    {
        Box::pin(async move {
            let core_id = convert::proto_to_core_resource_id(id);
            self.provider()
                .read(&core_id, identifier, CoreReadRequest)
                .await
                .map(|state| convert::core_to_proto_state(&state))
                .map_err(Self::convert_error)
        })
    }
}
```

`create`, `update`, `delete`, and `read_data_source` follow the same pattern: build core inputs, await the existing core provider future, convert success or error. There is no `tokio::runtime::Runtime`, no `new_current_thread`, and no `block_on`. Account-guard behavior does not move later; `initialize` still rejects wrong accounts before the host accepts the provider instance. AWSCC follows the same pattern, except its eager guard is represented by `AwsccProvider::new_with_config(...).await` plus `provider.init_error()`.

The normalizer methods are not part of the recommended async WIT set. If they remain sync WIT exports, AWS/AWSCC can keep the existing `runtime.block_on(self.normalizer...)` only if the runtime field still exists for non-I/O normalizer futures, which is undesirable. Prefer changing SDK normalizer hooks back to pure sync in the WIT-facing trait if they are pure, or include them in the async WIT set so they can await the existing core `ProviderNormalizer` futures without a runtime. This is an open implementation decision because current core normalizer traits are async-shaped even when their bodies are pure.

## Host call-site change

Current host code calls `bindings.call_initialize(&mut store, &wit_attrs).await`. After async-lifting an export, verify the generated method first. If wit-bindgen emits a high-level async method that already wraps `Store::run_concurrent`, use it and keep `WasmBindings::call_initialize` as the adapter. If it emits a task-returning or low-level function shape, implement the adapter explicitly:

```rust
store.run_concurrent(async |accessor| -> wasmtime::Result<_> {
    let result = provider_func.call_concurrent(accessor, params, results).await?;
    Ok(result)
}).await?
```

The exact generated names must be read from the generated binding compile errors or expanded code; do not guess. The wasmtime 45 source documents that `call_concurrent` futures must be awaited inside `run_concurrent` for progress, and that dropping the future does not cancel the guest task. This directly affects timeout and poison handling.

Store mutex handling has two layers. Keep the per-instance `Mutex<Store<HostState>>` held across a single guest operation because wasmtime still needs exclusive mutable access to the store for `run_concurrent` setup and result collection, and carina's `SharedWasmInstance` contract is one operation at a time per provider instance. Do not promise concurrent CRUD calls on the same store in this PR. Do remove the broader `shared_instances` mutex hold around initialization: `get_or_create_shared_instance` currently holds the map lock across `create_initialized_instance(...).await`. Change it to check the map, drop the map lock, initialize the instance, then reacquire to insert; if another task inserted the same key meanwhile, discard the new instance and use the existing one. This prevents one hung initialization from blocking unrelated binding keys at the factory map level.

For `create_initialized_instance`, apply the same hard timeout and poison semantics used by CRUD. A timeout during `initialize` leaves the new store undefined; drop it and do not insert it into `shared_instances`. Since no shared instance exists yet, this does not need `SharedWasmInstance::poisoned`, but the error should be the same `ProviderError::timeout` shape at the factory boundary. If initialization times out after the instance has been wrapped for uniform handling, mark it poisoned before dropping.

For CRUD, `LockedStore::acquire` should still set `store.set_epoch_deadline(WASM_OPERATION_TIMEOUT_SECS)` immediately before the operation. Epoch interruption remains the compute-time guard. `WASM_OPERATION_HARD_TIMEOUT` remains the wall-clock guard around the whole store lock plus guest call. Because `call_concurrent` cancellation is not per-task cancellation in wasmtime 45, a hard timeout must still poison the whole `SharedWasmInstance`; dropping only the future loses the host's result handle while the guest task may continue. The existing `PoisonOnDrop` field-order guarantee remains valid and should be preserved.

Async exports should reduce the H1 livelock by letting the component task return a future to the host runtime instead of blocking inside a sync task. They do not remove the need for host-side wall-clock timeouts. A slow AWS waiter can still exceed the backstop, and a cancelled wasmtime async task still leaves the store unsafe for reuse.

## PR sequencing across 3 repos

PR 1 lands in `carina`. It changes WIT, SDK, host, mock provider, generated bindings, and tests. CI must pass using the mock provider only. This PR cannot require new AWS or AWSCC provider commits to verify correctness because those repos depend on the SDK revision produced by this PR. It should include a clear compile-time break for old providers: components built against the old WIT fail instantiation with a WIT mismatch instead of silently using a legacy sync path.

PR 2 lands in `carina-provider-aws`. It re-pins `carina-plugin-sdk` and related carina dependencies to the merged carina revision, removes the guest runtime, converts the WIT-facing process provider methods to boxed futures, and rebuilds the WASM component. This PR depends on PR 1 because the new SDK trait and WIT files do not exist before PR 1 merges.

PR 3 lands in `carina-provider-awscc`. It performs the same migration for AWSCC. It also depends on PR 1 and can be developed in parallel with PR 2 after PR 1 is available, but it cannot merge and ship a compatible component until it pins the merged SDK revision.

The dependency edge is one-way: providers require the new carina SDK; carina's own PR verifies with `carina-provider-mock`. Do not make PR 1 wait for provider PRs, and do not merge provider PRs against an unmerged SDK SHA unless the release process explicitly supports temporary git rev pins.

`carina-rs/infra` consumers use provider `revision = 'main'` in many provider declarations. After PR 2 and PR 3 merge to provider `main`, `carina init` or a clean re-init resolves the new provider components. If a local `carina-providers.lock` already pins old provider SHAs, it must be refreshed. There is no DSL migration for infra; the timing risk is mixed binaries: a new carina host with old provider components should fail fast at load, and old carina with new providers should fail fast with a missing async WIT shape.

## Breaking-change posture

This is a breaking SDK and WIT change. Do not add a compatibility shim, do not keep a legacy sync provider interface, and do not translate old sync components at runtime. Any downstream provider that needs the old behavior pins carina and the provider SDK to a pre-fix revision. The host should report a direct error when a component exports the old sync interface so users know to rebuild or update the provider.

## Migration of carina-rs/infra

Infra users should see a provider revision refresh only. `provider aws`, `provider awscc`, aliases, `assume_role`, `allowed_account_ids`, `forbidden_account_ids`, region spelling, directives, resources, data sources, and state do not change. The expected operational sequence is: update the carina binary to a revision containing PR 1, wait for AWS and AWSCC provider `main` to contain PR 2 and PR 3, run `carina init` in the infra component directories so `carina-providers.lock` resolves the new provider components, then run `carina plan`. A stale lockfile is the main failure mode.

## Regression test plan

Once the parent stage C lands a hanging carina-only regression test, keep that test and change only the expected result: the same workload must complete within the bound after async-lifted exports. The best test remains a `carina-plugin-host` integration test that instantiates multiple WASM provider instances and performs enough HTTP work during `initialize` to match the assume-role credential-chain intensity. It should fail on the current sync export shape and pass after this design.

If stage C cannot make the mock hang, replace the weak mock loop with a stronger deterministic test. Two acceptable shapes are: a localhost STS-like server plus a small WASM guest that performs a base credential request followed by a signed assume-role request chain, or a stress test that starts many `wasi:http/outgoing-handler` requests from `initialize` with controlled delayed responses and verifies bounded completion under repeated runs. The current simple mock that issues a few plain HTTP requests is not enough because the parent investigation showed it passes while production still hangs.

Add compile-time or instantiation tests for the WIT break. A provider component built from the new mock should instantiate in both `carina-provider` and `carina-provider-with-http` worlds as appropriate. A fixture built against the old sync WIT, if practical to keep as a tiny checked-in component, should fail with a clear error. If keeping a binary fixture is undesirable, test the error formatting path using a component with the wrong export shape generated during the test.

Run crate-scoped verification first: `cargo test -p carina-plugin-sdk`, `cargo test -p carina-provider-mock`, and `cargo test -p carina-plugin-host`. Because the WIT and SDK changes touch cross-crate generated bindings, finish with `cargo test --workspace` only after the targeted tests pass.

## Risk and rollback

The largest risk is that mixed sync/async WIT exports are not usable with the pinned wit-bindgen/wasmtime combination even though the source tree contains async support. The rollback is to switch from Option C to Option A within the same design: mark the whole provider interface async and update all host calls consistently. That increases churn but keeps the root-cause direction.

The second risk is that wit-bindgen's guest export support requires an executor or task model that conflicts with the SDK's no-runtime goal. The rollback is not to restore sync exports. Instead, narrow the first implementation to async-lifted `initialize` only, prove the issue#3400 path, and then expand CRUD once the binding pattern is understood. This partial ship is only acceptable if no CRUD export can call `wasi:http` before the expansion PR; for AWS/AWSCC that is false, so partial shipping would be a short-lived branch, not a release.

The third risk is that wasmtime 45 still livelocks even with async-lifted exports because the generated guest code still calls blocking WASI poll APIs internally. If that happens, the rollback is to continue the async WIT change but also rewrite `wasi_http.rs` to await WIT future/pollable support instead of calling `pollable.block()`. The source of truth is the generated guest code and a reduced repro trace, not a guess about wasmtime internals.

If the implementation becomes too broad, the safe rollback is to abandon the PR before merge and leave the current branch unchanged. Do not merge a wasmtime bump, rustls initialization, or weak mock test by itself; the parent investigation already showed that combination does not fix the hang.

2. Does wasmtime 45's generated binding layer expose high-level async methods for async-lifted exports, or must carina call `Func::call_concurrent` inside `Store::run_concurrent` manually?
3. Does the carina host build with wasmtime 44 and async component exports, or is 45 the minimum practical version? The local registry used for this design had 45.0.1 sources but no 44 sources.

5. Should pure normalizer exports stay sync by changing the SDK WIT-facing normalizer contract back to sync, or should they be included in the async WIT set because the core `ProviderNormalizer` trait is already async-shaped?
6. Can `wasi_http.rs` be converted from `pollable.block()` to an awaitable WIT future with the pinned `wasi` and `wit-bindgen` versions, and would that be required for the fix or only a cleanup after async exports land?
7. What is the smallest mock or fixture that fails on the current sync export shape with the same NeedWork/StartImplicit pattern as the AWS assume-role repro?

## Implementation notes
<!-- derived-from #wit-level-design-choices -->

`async func` syntax worked directly with the WIT tooling pinned on this
branch. No WIT syntax fallback or wasmtime/wit-bindgen version bump was
needed for the schema itself, but guest export generation required the
`wit-bindgen` `async` feature; the SDK and mock wasm target dependencies
now enable `features = ["macros", "async"]`.

wasmtime 45 bindgen generated owned-argument async export methods that
take `&wasmtime::component::Accessor<_>` rather than `&mut Store<_>`.
The host adapters therefore wrap the six async exports in
`store.run_concurrent(async |accessor| { ... })` and pass cloned/owned
WIT arguments into the generated `call_*` methods. The cheap metadata,
validation, completion, identity, enum alias, and normalizer calls keep
the existing sync generated shape.

`build_engine_config` now enables `wasm_component_model_async(true)`.
`concurrency_support` is left at its wasmtime default (`true`), which is
the setting required for `Store::run_concurrent`.

The guest SDK uses `futures_util::lock::Mutex` for provider state in the
`export_provider!` macro. Async exports call `.lock().await`, while sync
exports call `try_lock()` and fail fast if a sync no-I/O export is invoked
while an async export is live. This avoids holding a `std::sync::MutexGuard`
across `.await` without pulling Tokio into wasm32.

`wasi_http.rs` remains unchanged. The mock's initialize/read/read-data-source/
create/update/delete implementations are ready futures around the existing
sync logic and the Stage C HTTP helpers.

`shared_instances` no longer holds the factory map mutex across
`create_initialized_instance`. The factory checks for an existing key,
drops the map lock while initialization runs, then reacquires the lock and
uses a concurrently inserted instance if one appeared first.
