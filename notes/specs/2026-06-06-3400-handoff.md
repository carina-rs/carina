## 一行要約

WIT export を async-lifted にする Stage A 改修を 4 repo / 7 PR で完走させ、すべて main に merge 済み。にもかかわらず実機の `carina-rs/infra/envs/registry/dev/infra` での `carina plan` は**依然として hang する**。wasmtime trace も Stage A 前と同じ `NeedWork ↔ StartImplicit` livelock を出す。Stage A の実装が実機に効いていないことが確定し、原因は未特定。

## 実機検証で起きていること

`aws-vault exec carina-registry-dev -- carina plan --refresh=false` を `carina-rs/infra/envs/registry/dev/infra` で実行すると:

1. 5 つの `Using ...` 行が出る
2. その後何も出ず hang
3. 240 s で kill しても完了しない
4. stderr は `Warning: using cached state` だけ。panic も error もない

`RUST_LOG=wasmtime::runtime::component::concurrent=trace` を付けると、Stage A 投入前 (wasmtime 43 時代) と**ビット同じ**の livelock パターンが出る:

```
new host task HostTask(3)
delete host task HostTask(3)
ready to delete? false ( ... host_future_state: Live)
suspend fiber: NeedWork
resume_fiber: suspend reason Some(NeedWork)
ready to delete? true ( ... host_future_state: Dropped)
delete guest task GuestTask(0)
queueing call QualifiedThreadId(0, 2)
push high priority: GuestCall(..., kind: StartImplicit)
... (永久ループ)
```

私が手元で aws/awscc の WIT export 本体を `runtime.enter()` から `runtime.block_on()` に書き換えた wasm をビルドして差し替えても同じ hang。WASM は `/tmp/carina-provider-aws-blockon.wasm` と `/tmp/carina-provider-awscc-blockon.wasm` にまだ残してある (再利用したければ)。

## 何が分かっていないか — 次の調査の起点

最も確度が高いのは順に:

1. **WASM artifact の export 形式を直接見ていない**。`wasm-tools component wit /tmp/carina-provider-aws-blockon.wasm` (もしくは infra に install された wasm) を叩いて、6 export が `async func` で出ているか sync `func` で出ているかを目で確認すること。これをやらずに私は Stage A を merge してしまった。もし sync で出ていたら、host bindgen 側だけが async expect で、guest WASM 側は sync export のまま、本来の bug が継続する。
2. **`shared_instances` Mutex 周辺の別の seam**。carina#3401 で hold 時間は短縮済みだが、`SharedWasmInstance` の構築は await 区間に残っている。management instance の init で他 binding が racing するパターンを再確認する。
3. **wasmtime 45 自体の bug**。PR #12043 は trap を入れただけで livelock を治していないかもしれない。upstream の wasmtime issue tracker に類似報告がないか追う。

mock-side の regression test (`multi_instance_wasi_http_initialize_*`) は H1 仮説の *shape* だけを catch しているもので、AWS-SDK の実コードパスが触発する pattern は再現できなかった。これは Stage C のノート末尾に書いてある通り。

## 設計面で私が間違えた所

- Stage A を「mock test pass + 手元 carina-plugin-host workspace verify green」だけで carina PR 化して merge した。**実機検証は provider PR merge 後**と思っていたが、それは間違い。少なくとも mock provider の wasm を `wasm-tools component wit` で dump して async export 形式を見れば、Stage A が WASM レベルで効くかは事前に判別可能だった。
- aws#408 / awscc#327 で `enter()` を採用したが、`enter()` は tokio runtime の handle を current にするだけで driver を pump しない。`tokio::time::sleep` は登録はされるが完了しない。私がレビューで気付くべきだった (CLAUDE.md の「短時間で exit した結果を成功と判定してはいけない」メモリと同種の判断ミス)。
- `block_on` 版を手元で試して同じ hang を確認した時点で、Stage A 自体に欠陥がある可能性に気付けた。

## 次セッションの初手 (推奨)

1. このメモと親 design note の "Post-Stage-A real-infra verification" 以降を読む
2. `~/.carina/cache` か `~/src/github.com/carina-rs/infra/envs/registry/dev/infra/.carina/providers/` から実際の aws/awscc wasm を取り出して `wasm-tools component wit <path>` で export 形式を dump する
3. 6 export が `async func` で出ていれば Stage A は WASM レベルで効いている → 原因は wasmtime 内部 (上記候補 3)
4. 6 export が sync `func` のままなら Stage A が guest 側で効いていない → 原因は wit-bindgen の async feature が CI build で有効になっていない / `export_provider!` macro の expand 結果に bug がある (上記候補 1)

次セッションでは推測でコードを書き換える前にこの WIT dump を見ること。私はそれを飛ばして 7 PR 走ったので user の hang が治っていない。

- carina-rs/carina: `387d64b5` (3401) と `0b1bdcd3` (3402) の 2 commit を revert

ただし、次の調査でもし「WASM の async export 形式は正しく出ている」と分かったなら、Stage A の WIT/SDK/host 改修自体は wasmtime 側の fix まで温存する価値がある。revert を急ぐ前に WIT dump で根拠を確認すること。
