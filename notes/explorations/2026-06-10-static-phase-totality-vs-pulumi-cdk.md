# 静的段階の全域性 — Carina と Pulumi / AWS CDK の比較

<!-- derived-from ./2026-06-10-type-system-type-theory-analysis.md -->

2026-06-10

このノートは、型システム分析レポートで土台として置いた「静的段階が全域的である」という Carina の性質を、Pulumi と AWS CDK という汎用言語埋め込み型のインフラ記述ツールと比較して整理したものである。結論を先に言うと、この性質は Pulumi / CDK では基本的に実現できない。実装の頑張りの問題ではなく、言語設計の選択そのものに由来する。

## 1. 「静的段階が全域的」とは何か

Carina は二段階の言語として読める。静的段階(validate / plan)で式の評価・型検査・効果列への変換を済ませ、動的段階(apply)で効果だけを実行する。このとき「評価が必ず停止する」(全域性)と「評価結果がソースと state だけの関数である」(純粋性)が言語の性質として成り立っているため、「プログラムを完全評価してから、出てきた値を全数検査する」が完全な決定手続きになる。型システム分析レポートで述べた値ベースの検査様式(elaboration 時に全数実行される契約検査)は、すべてこの土台の上に載っている。

全域性は言語を制限することでしか得られない。Carina の DSL に一般再帰はなく、for は有限コレクション上の展開しかできない。ただし構文制限は片方の柱にすぎない。相互参照(循環する値の流れ)というもう一つの発散源があり、こちらは構文では塞げないため、参照グラフの非循環性強制で塞ぐ。Carina では循環する let 束縛が評価前に静的拒否され、リソース間の依存循環は明示エラーになる(評価器は資源間の式を収束まで反復せず、未解決参照は動的段階が依存グラフの位相順で解決する)。つまり全域性は「構文制限 + 参照グラフの非循環性強制」の合わせ技であり、いずれにせよ後付けできる種類の性質ではない。

## 2. 汎用言語埋め込みが構造的に失うもの

Pulumi と CDK は逆の取引をしている。リソースグラフを構築する段階(Pulumi のプログラム実行、CDK の synth)は、TypeScript / Python / Go といったチューリング完全な汎用言語の任意のプログラム実行である。無限ループも、ネットワーク I/O も、環境変数や現在時刻への依存も書ける。したがって「グラフ構築は必ず停止する」「同じソースから必ず同じグラフが出る」は言語の定理としては成立せず、プログラマとライブラリの行儀に依存する経験則になる。

失われるのは停止性だけではない点に注意したい。Carina の plan がソースと state の純関数であるのに対し、synth の出力は外界に依存しうる(CDK の context lookup のように、synth 中に AWS へ問い合わせる仕組みは公式機能ですらある)。これを言語保証に戻すには汎用言語をサブセット化するしかなく、それはもう別言語を作ることと同じである。

## 3. 二段階の分離の度合い

ただし Pulumi と CDK を一括りにすると見落としが出る。二段階の分離がどこまで保たれているかは両者で違う。

CDK は構造的には Carina に近い。synth(任意プログラム)→ CloudFormation テンプレート(閉じた宣言的成果物)→ deploy という流れで、動的段階への入力は全域的な成果物になっている。壊れているのは「静的段階の停止性・純粋性」だけで、検査と実行の境界が具体的な成果物(テンプレート)に一致するという性質は保たれている。

Pulumi は分離自体が崩れている。`Output.apply()` のコールバックは依存リソースが実体化した後、つまり deploy の最中に実行される。プログラムの一部が動的段階で走るので、`pulumi preview` は構造的に近似でしかなく、コールバック内の分岐でリソースを作るようなコードは preview に現れないことがある。Carina の「plan は動的段階の静的近似であり、近似誤差は Unknown として明示される」という性質に対して、Pulumi は近似誤差の所在がプログラム中の任意のクロージャに分散する。

## 4. 未知値の意味論 — Unknown / Token / Output

段階を跨ぐ値(apply まで決まらない値)の扱いに、この設計差が最も鮮明に現れる。

Carina の Unknown は定義された意味論を持つ一級の値である。値の所属検査は葉の位置で素通し(周囲の構造検査は続く)、等価性は何とも成立しない、未解決参照には宣言型ベースの代入可能性検査が別経路で走る。未知値をどう扱うかが言語の意味論として固定されている。

CDK の Token は未解決値のプレースホルダだが、型上はただの `string` である。`if (vpcId === "...")` のような分岐や比較に使ってもコンパイルも実行も通り、プレースホルダ文字列への操作として黙って間違った結果を返す。CDK の有名な落とし穴であり、汎用言語に埋め込むと未知値の意味論をホスト言語の制御フローに守らせる手段がない、ということの帰結である。

Pulumi の `Output<T>` は CDK より規律が強い。型付きのコンテナなので、TypeScript 上で `Output<string>` を素の `string` として使えば型エラーになり、値に触るには `apply()` を経由するしかない。未知値の型レベルの規律という一点では Pulumi が三者で最も明示的とも言える。ただしその `apply()` こそが §3 で述べた「動的段階で走るコールバック」であり、規律の代価として段階の分離を失っている。

整理すると、CDK は段階の分離を保って未知値の規律を失い、Pulumi は未知値の規律を強めて段階の分離を失い、Carina は言語を小さくすることで両方を同時に手にしている、という三すくみになる。

## 5. トレードオフは相補的

公平のために、汎用言語埋め込みの見返りも明記しておく。TypeScript の型検査は、グラフを組み立てるメタプログラムに対する本物の項ベース静的型付けであり、関数・クラス・パッケージエコシステム(npm / PyPI)による抽象化もそのまま使える。型システム分析レポートの言葉で言えば、構図は綺麗に相補的である。

| | メタプログラムの項ベース型付け | 静的段階の全域性・値の全数検査 |
| --- | --- | --- |
| Pulumi / CDK | 強い(ホスト言語の型システム) | 持てない |
| Carina | ほぼ持たない | 持つ |

Carina 側の代償は言語が小さいこと自体である。全域的な DSL には汎用言語の抽象化手段(高階関数、再帰的なデータ変換、外部ライブラリ)がなく、表現力の上限は言語設計者が用意した構成子で決まる。

## 6. 系譜

Carina の選択は孤立したものではない。Terraform の HCL、Bazel の Starlark、Dhall、CUE はいずれも「全域的な設定記述言語」の系譜にあり、評価の停止を言語で保証した上で、評価結果に対する検査や合成を組み立てている。逆に Nix はチューリング完全であり、この系譜には入らない(Nix の評価は停止しないことがある)。

Pkl(Apple)を加えると、この系譜は一本の線ではなく二軸の空間として見えてくる。Pkl は専用の設定記述言語で、型注釈に任意の述語を付けられる(`port: Int(this > 0 && this < 65536)`)。検査はその述語を評価済みの値へ適用する方式で、意味論的型付け + 値ベースの契約検査という検査様式だけ見れば、この系譜の中で Carina に最も近い。ところが停止性の軸では反対側にいる。Pkl は再帰関数を書けるチューリング完全な言語で、評価は停止しないことがあり得る(CLI に評価タイムアウトのオプションがあるのはその裏返しである)。純粋性はモジュール・外部リソースへのアクセス許可制というサンドボックスで運用的に確保するが、全域性は言語の定理としては持たない。

| | 全域的 | チューリング完全 |
| --- | --- | --- |
| 専用言語 | HCL / Starlark / Dhall / CUE / Carina | Nix / Pkl |
| 汎用言語埋め込み | (構造的に不可能) | Pulumi / CDK |

「専用言語かどうか」と「全域的かどうか」は独立の選択であり、埋め込み+全域の象限が空なのは偶然ではない(ホスト言語をサブセット化しない限り埋まらない)。そして Pkl の位置は、全域性の価値が「評価結果が何をゲートするか」に依存することを照らす。Pkl の評価結果は JSON/YAML という成果物で、評価が止まらなければビルドが失敗するだけ — 世界は何も変わらない。一方 Carina や Terraform の静的段階は、実インフラを変更する動的段階の門番であり、門番が停止しないことは成果物生成の失敗より重い。Pkl が全域性を捨てて値ベースの篩検査だけを取れたのは、守るべき apply 段階(と、それに伴う unknown 値の問題)をそもそも持たないからだ、と読める。

つまり「静的段階の全域性」は、インフラ記述ツールの設計空間における一つの明確な分岐点である。汎用言語に埋め込んでエコシステムと項ベース型付けを取るか、専用言語を作って全域性と値の全数検査を取るか。Carina は後者を選び、型システム分析レポートで述べた検査様式の全体(意味論的型付け、篩の任意述語、型主導の等価性、Unknown の一貫した意味論)は、その選択の配当として読める。
